# MeshKit Architecture

[简体中文](../ARCHITECTURE.md) | English

MeshKit is an open-source P2P utility project based on WebRTC. The current version is `1.1.0`. The repository uses pnpm workspace; Web, Desktop, and signaling services share part of the core logic.

## Overview

```text
packages/web        packages/desktop
React + Vite        Electron + React
      |                    |
      +---------+----------+
                |
          packages/core
       P2P / file transfer
                |
                v
apps/signaling: WebSocket + PeerJS
                |
                v
       WebRTC DataChannel
       Browser to Browser
```

The signaling service handles device discovery, room state, and WebRTC negotiation. File contents, Notes Wall sync data, and chat messages are transferred through WebRTC channels whenever possible and are not stored long-term by signaling.

## Repository Structure

```text
apps/
  signaling/             WebSocket + PeerJS signaling service
packages/
  core/                  P2P, file transfer, shared types and utilities
  web/                   Browser application
  desktop/               Electron desktop application
docs/                    User and development documentation
docker-compose.yml       Web + signaling compose setup
```

## Main Modules

### packages/core

The core package provides shared cross-platform capabilities:

- PeerJS/WebRTC connection management.
- File chunks, queues, progress, and cancellation state.
- Device identity, transfer message types, and common utilities.
- Type definitions shared by Web and Desktop.

The core package should remain as runtime-agnostic as possible. Browser- or Electron-specific logic belongs in the app layers.

### packages/web

The Web app provides UI and browser runtime capabilities:

- File transfer page with peer-to-peer and pickup-code modes.
- Notes Wall page using Yjs and y-webrtc sync.
- Encrypted chat page.
- Share dialogs, QR codes, invite links, and mobile layout.
- favicon and MeshKit brand icons.

### packages/desktop

The Desktop app is based on Electron and currently supports Windows and macOS:

- The main process manages windows, local services, and system capabilities.
- The preload layer provides a controlled IPC bridge.
- The renderer provides Desktop UI and reuses part of the Web-side capability.
- The build directory stores icons and packaging resources for Electron Builder.
- Desktop can import invite links generated by Web and jump to the target page.

### apps/signaling

The signaling service is a lightweight Node service:

- WebSocket for device discovery and room messages.
- PeerJS server for WebRTC negotiation.
- `/healthz` for health checks.
- Periodic cleanup for stale devices and rooms.

## Runtime Flows

### Device Discovery

1. The client connects to signaling after startup.
2. The client registers device info and Peer ID.
3. Signaling broadcasts the online device list.
4. The sender selects a target device and starts connection.

### Peer-to-peer File Transfer

1. The sender selects files and a target device.
2. Both sides establish a WebRTC DataChannel through PeerJS.
3. The sender sends file metadata and chunks.
4. The receiver reassembles files and provides save actions.
5. The receiver marks completion, and the sender ends the transfer.

### Pickup-code Transfer

1. The sender creates a pickup room and a 6-digit pickup code.
2. The sender shares the pickup code, invite link, or QR code.
3. The receiver joins the room and establishes RTC.
4. The receiver saves files one by one.
5. Cancellation or completion is synced to the page state.

### Notes Wall and Encrypted Chat

1. Users create or join a room.
2. Room passwords are used for entry checks.
3. Room state is maintained through signaling and local state.
4. Notes content is synced through Yjs/y-webrtc.
5. When the owner destroys a room, members are notified and leave.

## Data and Storage

- File contents are not written to the signaling service.
- Web commonly uses localStorage, IndexedDB, or memory state for preferences, recent rooms, and collaboration cache.
- Desktop can use Electron main process and local filesystem capabilities to manage local service state.
- Invite links may contain room, connection, and sharing parameters and should be treated as sensitive.

## Ports

Default development and deployment ports:

| Service | Default Port | Purpose |
| --- | --- | --- |
| Web | 3000 | Browser entry |
| WebSocket | 7000 | Device discovery and room messages |
| PeerJS | 8000 | WebRTC negotiation |

Desktop built-in services prefer the default ports. If a local port is occupied, Desktop automatically tries later available ports. macOS AirPlay Receiver often uses `7000`, so the actual WebSocket port should follow the Desktop Share Hub display. LAN access requires firewall permission for the corresponding ports.

## Intended Boundaries

MeshKit is best for temporary collaboration between people who know each other, such as file transfer, notes, or chat within the same LAN.

- WebRTC provides transport-layer encryption.
- Room passwords control entry.
- Signaling does not store file contents.
- Do not publish invite links to uncontrolled environments.
- The current version does not provide accounts, audits, or complex permission systems.

## Extension Direction

Future extensions can follow the existing boundaries:

- Add new transfer message types in core.
- Add new collaboration pages in Web.
- Add tray, auto update, or richer local service management in Desktop.
- Add more detailed room lifecycle and deployment observability in signaling.
